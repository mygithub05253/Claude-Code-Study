# Task State Checkpoint

## Core Concepts / How It Works

### Information Stored in a Checkpoint

A checkpoint is not just a "note" — it structurally stores all the context needed to resume work.

```
Checkpoint snapshot structure:
  ├─ Git State
  │   ├─ Current branch name
  │   ├─ Latest commit hash + message
  │   ├─ List of staged/modified files
  │   └─ Whether there is a stash
  ├─ List of decisions made
  │   ├─ "Notice list uses SSR (reason: realtime not needed)"
  │   └─ "Image upload excluded from MVP"
  ├─ Remaining task list (checkboxes)
  │   ├─ [ ] Notice creation form validation
  │   ├─ [ ] Officer permission middleware
  │   └─ [ ] Pagination implementation
  └─ Current blocker / next first action
```

### When to Use

- Just before ending a session, when you want to record where to start next time
- When you need to switch context to another task mid-work (urgent bugfix, meeting, etc.)
- When you want to save today's progress during a multi-day task
- When you open a new session and want to quickly recall "where was I?"
- When handing off work between Conductor workspaces or branches

### Save vs Resume

```bash
# Save
> checkpoint  # or
> save progress  # or
> Let's stop here today, save checkpoint.

# Resume
> where was I  # or
> resume  # or
> what was I working on
```

### Storage Location

Checkpoints are saved in a memory file (`MEMORY.md` or a dedicated `checkpoint.md`) and persist even after the session is fully closed. In this project, they are saved under `C:\Users\kik32\.claude\projects\...\memory\`.

## One-Line Summary

This work state management skill **saves a snapshot** of the current Git state, decisions made, and remaining task list so that even if a session is interrupted or context switches, you can resume from exactly where you stopped.

## Learning Points / Common Pitfalls

- **The trap of "I'll remember it"**: When you come back 2 hours later or the next morning, you spend more time than expected trying to recall "where was I, why did I make that decision." Checkpoints reduce this recall time to nearly zero.
- **Save the reason for decisions too**: The key is not just "what was done" but also "why that decision was made." Losing the reasoning leads to re-examining the same issue later.
- **Make it a routine before ending sessions**: If you develop the habit of running checkpoint as the last command when finishing work, it naturally integrates with the learning records from the Learn skill.
- **Common mistake**: Ending the session without saving a checkpoint. In case the session is unexpectedly interrupted, save intermediate checkpoints during long work sessions.
- **Next.js 15 perspective tip**: If you have changes temporarily saved with `git stash` during feature branch work, it is good to also record the stash list in the checkpoint.

# 브레인스토밍 (Brainstorming)

## 한 줄 요약

기능/컴포넌트/동작 변경 등 **창작 작업 이전에** 사용자의 진짜 의도와 요구사항을 끌어내기 위한 구조화된 질문-탐색 스킬이다. 코드를 쓰기 전에 "왜/무엇을/어떻게"를 먼저 합의한다.

## 언제 사용하나요?

- 새 기능이나 컴포넌트를 만들라는 요청을 받았을 때
- "로그인 폼 개선해 줘" 같이 **해석의 여지가 큰** 요청이 들어왔을 때
- 기존 동작을 바꾸려 할 때 (리팩토링, 마이그레이션 포함)
- 작업 범위가 불확실해서 곧장 계획서(writing-plans) 단계로 넘어가기 어려운 경우

반대로, 타이포 수정이나 명백한 버그 픽스처럼 의도가 이미 분명한 경우에는 이 스킬을 건너뛴다. "이 함수 이름 바꿔 줘" 같이 구체적인 지시도 브레인스토밍 없이 바로 실행해도 된다.

## 핵심 개념

원본은 브레인스토밍을 "하드 게이트(HARD-GATE)"로 취급한다. 즉, 다음 단계(계획 작성/코드 작성)로 넘어가기 전에 반드시 통과해야 하는 관문이다. 핵심 아이디어는 네 가지다.

1. **의도 우선**: 무엇을 만들지가 아니라, 그 기능이 풀려는 문제를 먼저 이해한다.
2. **요구사항 분리**: 필수(must-have)와 선택(nice-to-have)을 사용자가 직접 구분하게 만든다.
3. **디자인 탐색**: 하나의 해법에 고정되기 전에 2~3개의 접근을 비교한다.
4. **명시적 합의**: 결과물은 "이제 writing-plans로 넘어가도 될 만큼 명확한가?"라는 질문에 yes가 나올 때까지 반복한다.

체크리스트적으로 보면, 브레인스토밍은 다음 흐름을 따른다.

- 문제 정의 → 페르소나/사용 시나리오 확인 → 성공 조건(Definition of Done) 도출 → 대안 비교 → 합의된 한 문장 요약

## 실전 예제 (대학생 관점)

**상황**: Next.js 15 App Router로 "동아리 공지 게시판"을 만드는 과제를 받았다. 교수님은 "CRUD 되고 로그인 있는 거면 된다"라고만 말씀하셨다. 곧장 `app/notices/page.tsx`를 만들기 전에 브레인스토밍을 돌려 보자.

```bash
# Claude Code 세션에서
> 동아리 공지 게시판을 Next.js 15로 만들려고 해. brainstorming 스킬로 먼저 의도부터 합의하자.
```

브레인스토밍 세션에서 Claude가 던지는 질문은 대략 이런 형태가 된다.

1. "공지의 핵심 독자는 누구인가요? (동아리원만 / 일반 학생도 포함)" → **인증 범위**가 달라짐
2. "공지를 작성하는 사람은 누구인가요? (회장만 / 임원진 / 모든 동아리원)" → **권한 모델**이 달라짐
3. "읽음 표시나 알림이 필요한가요?" → **DB 스키마**가 달라짐
4. "게시판 말고 Discord로 대체 가능한가요?" → **그냥 만들지 말지**를 결정

결과로 다음과 같은 한 문장이 나온다.

> "로그인한 동아리원(읽기)과 임원진(쓰기)이 사용하는, Next.js 15 + Supabase 기반의 공지 페이지. 알림은 MVP에서 제외."

이 문장만 있으면 이제 `writing-plans` 스킬로 넘어가 구체적인 라우트/테이블/컴포넌트 계획을 세울 수 있다.

```ts
// 브레인스토밍 이후에야 아래와 같은 결정을 자신 있게 내릴 수 있다
// app/notices/page.tsx — 서버 컴포넌트 (읽기 전용)
// app/notices/new/page.tsx — 임원진만 접근 (권한 미들웨어 필요)
// lib/auth.ts — 역할 기반 가드
```

## 학습 포인트

- **"코드 먼저"의 유혹에 저항하기**: 대학생 과제에서 제일 흔한 실수는 요구사항을 절반만 이해한 채 `npm create next-app`부터 치는 것이다. 브레인스토밍은 15~30분을 "말로 푸는" 데 쓰는 대신, 엉뚱한 방향으로 3시간 헤매는 걸 막아 준다.
- **질문 리스트를 외워 두기**: "누가 쓰는가 / 무엇을 원하는가 / 언제 실패하는가 / 왜 지금인가" 4가지만 기본으로 물어봐도 90%의 모호함이 해소된다.
- **Next.js 15 관점 팁**: 서버 컴포넌트 vs 클라이언트 컴포넌트 선택은 실제로는 브레인스토밍 단계의 결정이다. "상호작용이 필요한가? 폼이 있는가? 실시간 업데이트가 필요한가?"를 먼저 합의해야 `"use client"` 배치가 깔끔해진다.
- **"원본에서 명시되지 않음"**: 원본 SKILL.md는 구체적인 질문 템플릿을 고정해서 제공하지 않는다. 상황에 맞게 질문을 구성하는 것은 사용자와 Claude의 공동 책임이다.
